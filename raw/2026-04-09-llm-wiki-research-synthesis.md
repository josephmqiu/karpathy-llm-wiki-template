# Research Synthesis: LLM-Maintained Personal Wiki Patterns

## 1. Reactions, forks, and implementations of Karpathy's gist

- The gist dropped **April 3, 2026** and within days had 5,000+ stars, hit HN front page (158 pts, 45+ comments), and spawned a wave of forks and commentary. The core pattern is now widely referenced as "the LLM Wiki pattern" or "idea file."
- **Substantive implementations worth studying**: [MehmetGoekce/llm-wiki](https://github.com/MehmetGoekce/llm-wiki) (L1/L2 cache architecture, Logseq + Obsidian adapters, bootstrap script), [NicholasSpisak/second-brain](https://github.com/NicholasSpisak/second-brain) (Obsidian-first, clean folder layout with `/second-brain-ingest` + `/second-brain-query` + `/second-brain-lint` skills), [AgriciDaniel/claude-obsidian](https://github.com/AgriciDaniel/claude-obsidian) (adds `/autoresearch` and a "hot cache" session-continuity pattern), and [Ss1024sS/LLM-wiki](https://github.com/Ss1024sS/LLM-wiki) (multi-platform: CLAUDE.md, AGENTS.md, .cursorrules, .windsurfrules from one spec).
- **[LLM Wiki v2 gist](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2)** extends Karpathy's with production lessons: confidence scoring, supersession, retention decay, typed entity graphs, hybrid BM25+vector+graph search for wikis past ~100 pages.
- Karpathy's own reported result: one research topic grew to ~100 articles, ~400K words, without him writing the prose himself.

## 2. Obsidian + Claude Code workflows

- The dominant pattern: **put `CLAUDE.md` at the vault root**, run `/init` once to let Claude map the vault, then iterate. Obsidian released **official Agent Skills in January 2026** that teach Claude Code Obsidian's file formats (frontmatter, wikilinks, callouts).
- **Key pattern — "hot cache":** `wiki/hot.md` (or equivalent) holds a compressed summary of recent sessions so each new session doesn't replay context. Hierarchy is `hot.md` → `index.md` → domain sub-indices → specific pages (attribution: AgriciDaniel/claude-obsidian).
- **Key pattern — L1/L2 split** (MehmetGoekce): L1 = rules, gotchas, preferences, credentials auto-loaded every session (~10-20 files, git-ignored for secrets). L2 = the wiki proper, loaded on demand. Routing rule: *"Would the LLM making a mistake without this knowledge be dangerous or embarrassing? Then it belongs in L1."*
- **Work log convention** (Mauricio Gomes): `~/work log/YYYY/YYYY-MM/YYYY-MM-DD.md`, auto-created by Claude. Keeps operational journal separate from semantic wiki.
- Notable plugin: [claudian](https://github.com/YishenTu/claudian) embeds Claude Code directly inside Obsidian as a chat panel.

## 3. Existing PKM patterns adapted for LLMs

- **Zettelkasten + LLM**: [A-Mem paper](https://arxiv.org/abs/2502.12110) frames agentic memory explicitly as Zettelkasten-derived (atomic notes, dynamic indexing, typed links). Obsidian plugin [zettelkasten-llm-tools](https://www.obsidianstats.com/plugins/zettelkasten-llm-tools) adds embedding search and LLM-assisted link discovery. The tension: atomic notes suit LLM reading, but the *cognitive* benefit of Zettelkasten comes from the human writing them — a recurring critique.
- **PARA + LLM**: Still the most common top-level folder scheme people pair with CLAUDE.md. Typical layout: `00-inbox/`, `01-projects/`, `02-areas/`, `03-resources/`, `04-archive/`. Works, but most Karpathy-pattern implementations replace `03-resources/` with the wiki sub-tree (`concepts/`, `entities/`, `sources/`, `synthesis/`).
- **What's "stuck"**: entity/concept/source/synthesis as the standard page-type taxonomy, YAML frontmatter with `type`, `sources`, `related`, `confidence`, `updated`, and `## [YYYY-MM-DD] ingest | Title` as the log-line prefix.

## 4. Multi-topic vault organization

- Strong community consensus from experienced PKM folks: **don't go deeper than two folder levels**; use MOCs (Maps of Content) and wikilinks for cross-topic discovery instead of duplicating or nesting notes. A note can be linked from multiple MOCs without moving it.
- **Hybrid is the winning pattern**: folders for high-level separation (by type or by topic), MOCs for contextual views, tags/frontmatter for filterable properties. Topic-first pure hierarchies break down precisely because of cross-topic entities — exactly the user's concern.
- **Obsidian Rocks' FINVA system** (attribution: obsidian.rocks) goes type-first by workflow state: Fleeting, In-progress, Notes, Views, Archives — explicitly rejecting topic folders because "folders are binary." Names and links do the organizing.
- **Recommended synthesis for this user**: type-first at the top (`raw/`, `wiki/concepts/`, `wiki/entities/`, `wiki/sources/`, `wiki/synthesis/`, `wiki/mocs/`), with one MOC per topic in `wiki/mocs/ai-safety.md`, `wiki/mocs/cooking.md`, etc. Multi-topic entities live in `wiki/entities/` once and are linked from every relevant MOC. This is how NicholasSpisak/second-brain and Karpathy's own example structure it.

## 5. CLAUDE.md / agent-instruction examples

- Every substantive implementation ships a complete CLAUDE.md template — the best reference examples are in the repos listed in Q1. Common required sections: **role/identity**, **directory map with "NEVER modify raw/"**, **frontmatter schema** (`title`, `type`, `sources:`, `related:`, `created`, `updated`, `confidence`), **ingest workflow** (5 phases: analyze → scan affected → update 5-15 pages → quality gate → log+commit), **query workflow**, **lint workflow**, **naming conventions**, and **log format**.
- **Ingest ripple-effect rule** (Karpathy-original, echoed by everyone): a single source update should touch **10-15 wiki pages** — the source summary, affected concepts, affected entities, the index, the log. If it only touches 1-2 pages you aren't compounding.
- **"Compile-first" rule** (Ss1024sS): every decision must be written back to the wiki, not just delivered as an answer. Wiki pages are the unit of truth; chat responses are ephemeral.
- **Platform-agnostic approach worth stealing**: keep CLAUDE.md, AGENTS.md, .cursorrules, and .windsurfrules in sync pointing at a single source-of-truth workflow doc, so the vault isn't locked to one agent.

## 6. Pitfalls and lessons learned

- **Error compounding** (HN consensus): a mistake in an early ingest gets backlinked into later pages. Human review of lint output is non-optional where factual accuracy matters. Karpathy himself recommends **ingesting one source at a time and staying involved**.
- **Source granularity matters enormously**: the most successful HN reporter split 3 books into chapter-level files and got 210 concept pages + 4,597 cross-references + 20+ cross-book syntheses. Naive whole-book ingests produced "slop."
- **Markdown is not a database** ([limitededitionjonathan](https://limitededitionjonathan.substack.com/p/stop-calling-it-memory-the-problem)): five failure modes — no querying, no relationship traversal, no concurrent writes, inconsistent schema, scale limits. Mitigation most implementations land on: enforce frontmatter strictly, lint often, and accept the scale ceiling (~100-300 sources) before you need Dataview, SQLite, or a graph DB.
- **Cognitive offloading** (multiple HN comments + Stack Overflow post): the act of writing is part of the thinking. Wikis that the LLM writes *entirely* on your behalf bypass the mental-model-building that made note-taking useful. Effective users treat Claude as the bookkeeper, not the thinker — humans still read/edit syntheses.
- **Context window is not a replacement**: degradation starts around 200-300K tokens even in 1M-context models, so the wiki-in-context shortcut doesn't scale indefinitely.

## Notable disagreements / open debates

- **"Is this just RAG?"** — the strongest critique and the strongest defense. Defenders: raw RAG returns chunks; a compiled wiki returns *structured, cross-referenced, deduplicated* knowledge. Detractors: it's RAG with extra steps that will be obsolete once context windows grow.
- **Model collapse vs compounding gains**: Shumailov et al. 2024 Nature paper on collapse was cited repeatedly; the counter was that well-curated LLM-on-LLM-output cycles have in practice delivered large wins.
- **Human in the loop, how much?** — Karpathy's "one source at a time, stay involved" vs fully-automated pipelines with scheduled ingest hooks (LLM Wiki v2 direction). No consensus; depends on factual-accuracy stakes.
- **Markdown-only vs markdown + DB**: split on whether a SQLite/Kuzu sidecar is worth it. Most personal-scale users say no for now; power users building for 1000+ sources say yes.

## Concrete patterns worth stealing (attributed)

1. **Folder layout**: `raw/` (immutable) + `wiki/{concepts,entities,sources,synthesis,mocs}/` + `wiki/index.md` + `wiki/log.md` + `CLAUDE.md` at root (Karpathy + NicholasSpisak)
2. **Hot cache file** `wiki/hot.md` for session continuity (AgriciDaniel)
3. **L1/L2 split** for rules vs knowledge, with L1 git-ignored for secrets (MehmetGoekce)
4. **10-15 page ripple rule** per ingest (Karpathy)
5. **Log prefix** `## [YYYY-MM-DD] ingest | Title` (Karpathy)
6. **Frontmatter**: `type`, `sources`, `related`, `created`, `updated`, `confidence` (consensus)
7. **One MOC per topic** in `wiki/mocs/` for multi-topic organization — entities live once, linked from many MOCs (Obsidian community consensus + second-brain repos)
8. **Chapter-level splitting** of long sources before ingest (HN success story)
9. **Ingest one source at a time** for anything factually load-bearing (Karpathy)
10. **Multi-agent files** (CLAUDE.md + AGENTS.md + .cursorrules) pointing at one shared workflow doc (Ss1024sS)
